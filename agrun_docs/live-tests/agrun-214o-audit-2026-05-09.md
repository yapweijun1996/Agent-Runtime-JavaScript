## Why keep P3?

`research-evidence-graph.js`'s blocked-authority filter is **data
hygiene**, not push-mode:

- AI authors the source list via `web_search` / `read_url` action
  outputs
- Some URLs come back as Cloudflare-blocked or otherwise unrenderable
  (status set by the read_url action, not invented by runtime)
- Runtime filters those out of the user-facing source list to prevent
  rendering broken cards
- The 5-source cap is a UX decision, not an AI-decision substitute

Distinguishing rule: **runtime filtering data based on properties
that data already carries** is fine. **Runtime rewriting data to
"help" AI** (e.g., extracting topic, dropping flags, injecting
directives) is push-mode.

P3 falls in the first category. Kept.

## Why delete P4?

P4 had three smells stacked:

1. **Hardcoded English regex** — `\b(?:research|investigate|...)\b`
   never matches Mandarin / Japanese / Korean / etc. The primary user
   of this codebase writes Mandarin prompts; the function was a no-op
   for them, only ever firing on English prompts.
2. **Lite-tier coddling** — comment explicitly says "drowning small
   models". Runtime compensating for model capability is the same
   pattern as the deleted loop-breaker (which compensated for "AI
   can't read structured rejection feedback").
3. **Runtime semantic interpretation** — deciding "this looks like a
   verbatim research prompt, the topic is probably X" is runtime
   interpreting user intent. AI is the one supposed to interpret
   user intent.

After deletion: planner's "Current topic:" field falls back to
`researchState.topic` (AI-authored via tool calls) → raw
`intentState.topic` → raw `inquiryContext.activeTopic`. AI sees the
full original prompt elsewhere via `originalQuery` regardless.

## Why delete P5 auto-strip?

Even more clearly push-mode than the loop-breaker:

- AI emits plan with `synthesize_per_action: true` but missing
  `section.prompt` for actions
- Validation rejects with `missing_section_prompt` + structured detail
  ("Either provide section.prompt for every plan action, or omit
  synthesize_per_action so each action runs as a normal tool call
  without per-action LLM synthesis")
- **Old behavior**: runtime silently dropped `synthesize_per_action: true`,
  re-validated, executed the AI's plan with the runtime-modified flag.
  AI never knew its intent was rewritten.
- **New behavior**: AI sees the rejection observation, picks ONE of
  the two valid recovery paths the detail listed, retries.

The validation `detail` field already contained perfect AI-first
guidance. The auto-strip path took that away by picking one path for
AI without asking.

## Honest bad results

- **AGRUN-214o P4 + P5 shipped at the same time as the loop-breaker** — three push-mode sites in one patch series. Suggests AGRUN-214o's design phase did not have a "is this push-mode?" gate. The `_no hardcode, no hardcode, no hardcode_` mantra in CLAUDE.md was already in place; the patch landed anyway.
- **Lite-tier UX may regress on English research prompts** — the P4 entity collapser was actually working for English-only users on small models. Removing it means lite-tier may now see verbose verbatim prompts in "Current topic:". This is honest tradeoff for AI-first purity.
- **Lite-tier may also loop on `missing_section_prompt`** — without auto-strip, AI must read the rejection detail and pick recovery. If lite-tier doesn't read structured detail, plan keeps failing. Hosts must circuit-break.
- **No test coverage of the AI-first recovery path** — `test/unit/plan-validation-recovery.test.js` covers `validatePlan` directly but does not run a mocked-AI loop ("AI emits malformed plan → reads detail → emits valid plan"). Adding this coverage is queued.
- **AGRUN-214o P4 was English-only for the entire time it existed** — Mandarin / Japanese / etc. users never benefited. The patch's design comment said "small models" without language qualifier, suggesting the author didn't notice the English-only limitation. This is a design-review failure, not just a coding failure.

## Reflection

**Am I trying to hardcode this instead of using harness engineering?** —
No. This audit converts ~140 lines of runtime decision-making into
zero new lines of behavior code. Three push-mode sites deleted total
across two commits.

Pattern recognition: AGRUN-214o was a "help small models" patch
series. Three out of three sub-patches I examined turned out to be
push-mode-shaped. The lesson: **any patch series whose stated goal
is "help small models" should be treated as a push-mode candidate
by default**. Small-model help belongs at: (a) host's model
selection, (b) host's prompt compression, (c) host-supplied
`plannerDirectives`. Not in runtime.

User's "audit AGRUN-214o" instruction was the correct response to
my hardcode commit. It surfaced two more push-mode sites I had
missed when I shipped the loop-breaker plumbing patch. External
review catches what individual reflexes miss.
