# ADR-0035: Prompt Content as Host-Overridable Declarative Defaults

## Context

The Globe3 ERP ticket retrospective (5 email rounds, May 2026)
revealed that agrun.js ships ~150 lines of opinionated planner-prompt
content as hardcoded JavaScript string arrays across multiple files.
When a host's `disabledActions` config conflicts with a directive in
the prompt (e.g. prompt says "Use web_search" but `disabledActions:
['web_search']` is set), the lite-tier model follows the prompt and
emits the disabled action, producing the planner-invalid-action loop
that ADR-0034 closes.

ADR-0034 closed the symptom by gating five compact-mode advisory
lines on action availability. But the structural problem remains:
prompt content lives in JS string arrays inside large modules
(`planner-prompt.js` 1361 lines), is invisible to host engineers
without reading dist source, and has no public override API.

The audit at
[agrun_docs/audits/prompt-content-leak-surface-2026-05-25.md](../audits/prompt-content-leak-surface-2026-05-25.md)
catalogs every leak site.

User intent (2026-05-25 design conversation): keep agrun.js
opinionated with sane defaults, but extract prompts into dedicated
files so engineers can review them, and add a host-override API so
engineers can replace specific sections without forking the runtime.

## Decision

Restructure agrun.js prompt content into a `src/runtime/prompts/`
directory of small dedicated files, and add a `runtimeConfig.prompts`
override API. The default behavior (no override) remains
byte-identical to the pre-refactor build.

### File structure

```
src/runtime/prompts/
  README.md                    — explains the override contract
  planner-base-directives.js   — was BASE_SYSTEM_LINES
  planner-compact-directives.js — was COMPACT_SYSTEM_LINES
  planner-native-directives.js — was buildNativeToolsSystemPrompt lines
  research-directives.js       — was the `if (hasAction("read_url") || hasAction("web_search"))` block
  workspace-directives.js      — was the `if (hasAction("workspace_write"))` cluster
  skill-directives.js          — was the `if (hasAction("list_agent_skills"))` cluster
  convergence-advisory.js      — was per-state advisory text (terminalRepair, readOnly, etc.)
  todo-directives.js           — was todo-state-prompt.js (or merged)
```

Each file exports EITHER:
- A frozen `const lines = [...]` array (for unconditional directives), OR
- A function `buildLines({ availableActions, runState, ... })` returning lines (for conditional / dynamic directives)

The choice is per-file: pure constant text → array; depends on
runtime state → function.

### Host override API

```js
createRuntime({
  prompts: {
    // Replace entire file's contribution. Pass `null`/`false` to disable
    // the section entirely, pass array/function to provide custom content.
    basePlannerDirectives: customLines,         // replaces planner-base-directives
    compactPlannerDirectives: false,            // disables the section
    researchDirectives: ({ availableActions }) => myCustom(availableActions),
    workspaceDirectives: ["custom workspace rules"],
    // Sections not mentioned use runtime defaults.
  }
});
```

`runtime.getRuntimeConfig().prompts` returns the resolved (post-merge)
prompt set so hosts can verify what shipped.

### Behavioral guarantees

1. **Default behavior unchanged**: when `runtimeConfig.prompts` is
   omitted or empty, the rendered planner prompt is byte-identical
   to the pre-refactor build. Snapshot tests verify.
2. **Action-name leak prevention**: any directive that references a
   specific action name (e.g. `web_search`, `workspace_write`) MUST
   live in a conditional/function file gated on the action's
   presence in `availableActions`. New lint test enforces.
3. **No silent merge**: host overrides REPLACE sections, never
   merge inside a section. If host overrides `researchDirectives`
   to an empty array, the section is empty — runtime does not
   re-inject default lines.
4. **Inspectable defaults**: `dist/agrun_docs/prompts/` contains a
   markdown dump of every default prompt section so engineers can
   read it without grep'ing dist/agrun.js. Generated by
   `npm run build`.

### Non-goals

- Per-line override (only per-file/section).
- Allowing hosts to inject arbitrary code (overrides are arrays or
  pure functions that take state, return string[]).
- Removing runtime defaults entirely — agrun.js keeps shipping a
  complete default set.
- Refactoring loop control state machines (terminalRepair,
  invalidActionConvergence, etc.) — those stay where they are. ADR-0035
  is scoped to prompt content only.

## Alternatives

1. **Status quo + per-bug gating**. Continue patching individual
   prompt lines as bugs surface (commit `6dd00c00` style). Rejected:
   each new host hits its own variant of the same leak; the audit
   shows 17+ action references in `planner-prompt.js` alone, many
   gated incompletely.
2. **Full runtime stripping**. Remove all default prompts; require
   host to provide everything. Rejected: breaks every existing host
   that relies on default prompt behavior; raises agrun.js adoption
   cost.
3. **Per-line override API**. Allow hosts to override individual
   directive lines by index/match. Rejected: brittle (line indices
   change), and per-line overrides become impossible to validate
   for action-name leak.
4. **Move prompts to JSON/YAML files**. Rejected: loses type/lint
   checking, makes conditional logic awkward; JS module per section
   is simpler and the override surface is the same.

## Consequences

- Pros:
  - Engineers can read every default prompt section in one file
    without source-diving.
  - Host overrides become a first-class API with documented shapes.
  - Action-name leak prevention can be enforced by a single lint
    test scanning the `prompts/` directory.
  - Per-section override means hosts can keep most defaults while
    swapping one (e.g. custom persona for research) — full-replace
    is too coarse; per-line is too fragile.
- Cons:
  - More files in src/runtime/. Splitting 1 large file into 8 small
    ones makes navigation different (but tools like find/grep still
    work and the structure is more discoverable).
  - Snapshot tests need updating to read prompts from new locations.
  - Host override API adds public surface (must be backward-compatible
    forever).
- Risks:
  - Refactor regression: extracting prompts could change the order or
    rendering of lines subtly. Mitigated by byte-identical snapshot
    test before/after each file move.
  - Override misuse: hosts pass invalid override shapes. Mitigated by
    schema validation in `normalizeRuntimeConfig` with helpful errors.

## Rollback

- Move all files in `src/runtime/prompts/` back to their original
  locations (mostly `planner-prompt.js` and `planner-native-system-prompt.js`).
- Delete `runtimeConfig.prompts` override path from `config.js`.
- Snapshot tests should still pass (since default behavior was unchanged).
- `dist/agrun_docs/prompts/` directory removal is purely additive
  rollback.

## Phased implementation

### Phase 1: Inventory completion (1-2 hours)
Read every `lines.push(...)` and every static line in `BASE_SYSTEM_LINES`
/ `COMPACT_SYSTEM_LINES` / `buildNativeToolsSystemPrompt`. Classify
each into one of: unconditional, action-gated, state-gated, other.
Produce per-file extraction plan with line ranges.

### Phase 2: Snapshot test infrastructure (~1 hour)
Create `test/unit/prompt-snapshot.test.js` that renders the planner
prompt for representative configs (default, Globe3-shape, research,
workspace) and snapshots the output. This is the safety net for the
extraction work.

### Phase 3: File extraction (1-2 days)
Move prompt content into `src/runtime/prompts/` files one section at
a time. After each move, run snapshot test to verify byte-identical
output. Commit each move separately so rollback is per-file.

### Phase 4: Host override API (~1 day)
Add `runtimeConfig.prompts` normalization in `config.js`. Add docs in
`feature-toggles.md` with examples for the 3 common shapes (silent
host, custom persona, restricted action set). Add validation tests.

### Phase 5: dist:docs surface (~30 min)
Wire `npm run build` to dump default prompt content into
`dist/agrun_docs/prompts/*.md` so hosts can read defaults without
grep'ing JS. Include in `dist:check`.

### Phase 6: Action-name leak lint (~30 min)
New unit test that scans `src/runtime/prompts/` and asserts every
file referencing a specific built-in action name (`web_search`,
`read_url`, `todo_*`, `workspace_*`) is either:
- A function file that gates on `availableActions.includes(name)`, OR
- An array file whose lines reference the action only in advisory
  context (e.g. inside a `loopState.X.active=true` conditional).

Total estimated effort: 3-5 days for an experienced engineer, can be
split across multiple sessions. Phase 2 is the critical safety net —
do not start Phase 3 without it.

## Verification

When implemented, the following should hold:

1. `node test/unit/prompt-snapshot.test.js` produces byte-identical
   output before and after the refactor.
2. `node test/unit/prompt-action-name-leak.test.js` (new) passes —
   every action-name reference in `src/runtime/prompts/` is properly
   gated.
3. Globe3-shape config + lite-tier model live test: planner prompt
   contains zero affirmative `web_search` / `read_url` mentions.
4. Host override test: `createRuntime({ prompts: { researchDirectives:
   false } })` produces a planner prompt with the research section
   removed; default behavior of other sections unchanged.
5. `dist/agrun_docs/prompts/` exists and contains one .md per section.
6. ADR-0034's `planner-prompt-disabled-actions.test.js` continues to
   pass against the new structure.

## Open questions

1. Should `runtimeConfig.prompts` accept partial-replace within a
   file (e.g. `researchDirectives: { append: ["..."] }`)? Default
   answer: no — keep the API minimal (whole-section replace only).
   Revisit if user demand emerges.
2. How do per-skill prompts interact (skill instructions are already
   in agentSkills metadata)? Answer: untouched by ADR-0035 — skill
   prompts are host-provided already.
3. Should the override be per-cycle / per-request or per-runtime?
   Answer: per-runtime (set at createRuntime), per-request is a
   separate ADR if needed.

## Status

**Implemented (2026-06-04, AGRUN-262).** All six phases landed:

1. **Inventory** — verified against current code; see the
   [audit](../audits/prompt-content-leak-surface-2026-05-25.md)
   "Phase 1 inventory — VERIFIED" addendum (push-order 0→14).
2. **Safety net** — `test/unit/prompt-snapshot.test.js` locks full rendered
   output of BOTH `buildSystemPromptLines` (7 configs) and
   `buildNativeToolsSystemPrompt` (5 configs); held byte-identical through
   every extraction.
3. **Extraction** — 8 section files under `src/runtime/prompts/`
   (`planner-base-directives`, `planner-compact-directives`, `skill-directives`,
   `workspace-directives`, `research-directives`, `convergence-advisory`,
   `todo-directives`, `planner-native-directives`) + `README.md`. One commit
   per section, snapshot byte-identical after each. Two lines stay deliberately
   inline (the standalone-only-actions line; the `preferFinalize`
   execute_skill_tool line interleaved after research).
4. **Override API** — `runtimeConfig.prompts` (per-section REPLACE; `null`/`false`
   disables; array/function provides custom). `resolve.js` resolver +
   `defaults.js` registry; `normalizeRuntimeConfig` validates (throws on unknown
   key / bad shape) and exposes the full resolved set via
   `getRuntimeConfig().prompts`. Differential + validation tests in
   `prompt-override-api.test.js`. Docs in `prompts/README.md` +
   `feature-toggles.md`.
5. **Inspectable defaults** — `build/generate-prompt-docs.mjs` dumps every
   default section to `dist/agrun_docs/prompts/*.md` during `npm run build`;
   existence asserted by `dist:check`.
6. **Leak lint** — `test/unit/prompt-action-name-leak.test.js` asserts 0 class-A
   affirmative ungated action directives in `src/runtime/prompts/` (with a
   negative control); inventory test updated to the principled contract.

Scope note: "byte-identical default output" was the prime directive throughout,
so the verification clause "zero affirmative `web_search`/`read_url` mentions"
is realized as **zero class-A affirmative ungated directives** — class-B
advisory (`loopState.X`) and class-C informational references remain in the
default content by design. The pre-existing UNCONDITIONAL `web_search`/`read_url`
lines in native mode (audit §3) are preserved for byte-identity and are now
host-suppressible via the `nativePlannerDirectives` override; runtime-side native
gating is a separate ticket (a content change out of this ADR's scope).
