# agrun.js

`agrun.js` is a browser-first agent runtime library written in vanilla JavaScript. Both browsers and Node.js (>= 18) are supported host environments.

The runtime core is environment-neutral: code shipped in `dist/agrun.js` must not depend on Node-only APIs, so the same bundle runs in a browser `<script>` tag, through a bundler, or in a Node.js process (`require("./dist/agrun.js")`). Node-specific capabilities — filesystem session storage, an SSE event-stream adapter, read-only repo file tools — are opt-in adapters under [node/](https://github.com/yapweijun1996/agrun/blob/main/0_development/node/README.md). Node.js is also used for local build, test, and live checks.

The repository keeps two truths in view:

- `src/` is the source of truth for implementation.
- `dist/agrun.js` is the shipped single-file bundle and must stay in sync with source changes that affect the bundle.

## Repository Layout

```text
src/                  runtime source and built-in skills
agrun_docs/           architecture, contracts, process, and ADRs
dist/                 committed release bundle (agrun.js + README.md + FLOWCHART.md + agrun.md + example/ static demo)
examples/browser/     browser example application
test/                 smoke and live test entrypoints
skills/               bundled local agent instruction packages
roles/                bundled agent roles (ROLE.md persona definitions)
scripts/              developer-facing helper scripts
```

`dist/README.md` is emitted by `npm run build` as the first-read project overview for consumers receiving only `dist/`. `dist/FLOWCHART.md` is emitted beside it as the visual runtime logic map. `dist/agrun.md` is auto-generated by `npm run build` as the usage navigation guide alongside `dist/agrun.js`; it links to `agrun_docs/usage-quickstart.md`, `agrun_docs/feature-toggles.md`, `agrun_docs/public-runtime-api.md`, and `agrun_docs/result-schema.md` with a build id and rewritten relative links — so consumers do not need to clone the repo to know how to use the runtime.

## Local Development

Install root dependencies:

```bash
npm install
```

Run the required root verification:

```bash
npm run check
```

That command builds the full `dist/` tree (lib + example), runs the smoke test suite, and runs the deterministic mock-transport repro gates (`npm run test:repro` — no API keys, no cost; see [test/livekit/README.md](https://github.com/yapweijun1996/agrun/blob/main/0_development/test/livekit/README.md)).

To generate a demo-ready distribution artifact without the full smoke suite:

```bash
npm run dist
```

That command runs the full build and `dist:check`, producing `dist/agrun.js` plus the static browser demo at `dist/example/`.

`npm run build` is a three-stage pipeline (see [agrun_docs/distribution-bundle.md](./agrun_docs/distribution-bundle.md)):

1. `npm run build:lib` — rollup builds `dist/agrun.js`, `dist/README.md`, `dist/FLOWCHART.md`, `dist/agrun.md`, `dist/agrun_docs/`, `dist/CHANGELOG.md`.
2. `npm run build:browser` — vite builds the React example into `examples/browser/dist/`.
3. `node build/copy-browser-dist.cjs` — copies the browser dist to `dist/example/` and runs a secret-leakage guard against env-injected API keys.

The browser example sub-project still has its own `npm install` requirement on first checkout:

```bash
cd examples/browser
npm install
```

For local interactive development:

```bash
cd examples/browser
npm run dev
```

Browser example notes:

- Enter provider keys and model ids in the browser Settings panel, or opt into repo-root `.env.local` auto-seeding as described in [agrun_docs/development-secrets.md](./agrun_docs/development-secrets.md).
- Provider-backed turns expose raw provider `usage` plus normalized token counts in the Inspector.
- Runtime planner config is visible in the Inspector and Support Bundle as `plannerMode` / `nativeToolsFailurePolicy`, so default native-tools turns can be checked without reading IndexedDB manually.
- Open the browser Inspector after a provider turn to see `Token Usage` in Summary and the raw `usage` payload in `Raw -> Response`.
- For repeatable QA, open the example with `?debug_yn=y&qa=<name>&qa_clean=y` to create a clean QA chat while preserving provider settings. The scripted live multi-turn check is `npm --prefix examples/browser run test:live:browser-multiturn -- --provider all`.

## Development Rules

- Keep runtime behavior environment-neutral (browser-safe and Node-safe). Do not add Node-specific runtime assumptions to code shipped in `dist/agrun.js`; Node-only capabilities belong in opt-in adapters under `node/`.
- Keep files small and single-purpose. Prefer refactoring before a file grows past the repository guidance in `AGENTS.md`.
- Resolve documentation conflicts through [agrun_docs/architecture-ssot.md](./agrun_docs/architecture-ssot.md): `task.jsonl` owns current planning (the structured task board), ADRs own durable architecture decisions, contract docs own host-facing behavior, and live-test records are historical evidence.
- Keep runtime boundaries explicit. Capabilities belong in skills when they do not require runtime-owned coordination such as planning, approval, session continuity, or evidence handling.
- If a change affects the built output, commit the updated `dist/agrun.js` in the same PR.
- If a change alters a public contract or architecture boundary, add or update an ADR and the related docs in the same PR.

## Build Artifact Policy

This repository intentionally commits `dist/agrun.js`, `dist/README.md`, `dist/FLOWCHART.md`, and `dist/agrun.md` so consumers without a Node toolchain can vendor a known-good release.

Update `dist/agrun.js` in the same PR when a change affects the shipped bundle, including changes to:

- files under `src/`
- bundled skill content under `skills/`
- bundle wiring in `build/`
- `rollup.config.js`
- entrypoint or export wiring that changes bundle output

`dist/example/` is regenerated on every `npm run build` from `examples/browser/`. It is the static demo path engineers use to evaluate agrun without integrating it. Whether to commit `dist/example/` is a release-strategy choice — see [agrun_docs/distribution-bundle.md](./agrun_docs/distribution-bundle.md). The secret-leakage guard inside `build/copy-browser-dist.cjs` aborts the build if any API key from `.env.local` ends up baked into the example bundle.

Docs-only changes do not require rebuilding `dist/agrun.js`.

## Multi-tenant Deployment Pattern

When multiple end users may share the same browser profile (e.g. a shared ERP workstation), isolate per-user state at the storage layer by scoping the IndexedDB database name with the authenticated user id. Do not rely on row-level filtering — sessions, messages, and global memory all live in the same database, so the only clean isolation boundary is the database itself.

```js
import { createIndexedDBSessionStore } from "agrun";

const sessionStore = createIndexedDBSessionStore({
  dbName: `agrun-${userId}` // server-validated userId from the host auth layer
});
```

On logout or GDPR erasure, drop the per-user database:

```js
indexedDB.deleteDatabase(`agrun-${userId}`);
```

Configurable global memory thresholds are available via `createRuntime({ globalMemory })`:

```js
createRuntime({
  globalMemory: {
    enabled: true,      // set to false to disable cross-session memory entirely
    minConfidence: 0.7, // floor for auto-promotion
    maxEntries: 100     // LRU cap per database
  }
});
```

Pass `globalMemory: { enabled: false }` to skip the per-turn semantic extraction LLM call, the recall read, and all promotion writes. Per-session memory keeps working; only the cross-session path is bypassed. Existing rows in IndexedDB are left untouched. See [agrun_docs/public-runtime-api.md](./agrun_docs/public-runtime-api.md#disabling-global-memory) for the full contract.

Host apps can observe global memory activity through the existing `onStep` callback by filtering on step types `global-memory-recalled`, `global-memory-written`, and `global-memory-purged`.

The `sessionStore` interface also exposes `updateGlobalMemory(id, patch)`, `deleteGlobalMemory(id)`, `clearAllGlobalMemory()`, and `readAllGlobalMemory()` for building a "manage my memory" settings UI.

## Verification Summary

- Required for normal PRs: `npm run check` (which runs `npm run build` — lib + example + secret guard — then `dist:check`, smoke tests, and the deterministic `test:repro` gates).
