### C1 — Dangling Promise After Action Timeout
- **File**: `src/runtime/action-execute-timeout.js`
- **Risk**: Unhandled rejection crashes Node.js 15+ process
- **Root cause**: `Promise.race([executePromise, timeoutPromise])` — when timeout wins, losing `executePromise` has no `.catch()` attached. Sibling `host-hook-timeout.js:40` already uses `hookCall.then(NOOP, NOOP)` — pattern exists, not applied here.
- **Fix**: In `.finally()` add `executePromise.then(undefined, () => {});` — mirrors existing NOOP pattern.

### C2 — Async onCheckpoint Hook Silently Unhandled
- **File**: `src/runtime/action-loop-session-loop.js` (emitCheckpoint ~line 703)
- **Risk**: Host async checkpoint handlers produce unhandled rejections → crash Node.js or silent browser error
- **Root cause**: Synchronous `try/catch` around async hook call. Promise return is discarded. `mergeHook` in `default-run-options.js` also discards the default hook's Promise.
- **Fix**: `const r = session.onCheckpoint(...); if (r?.then) r.then(undefined, () => {});`

### C3 — executePlan Has No try/catch
- **File**: `src/runtime/action-loop-plan.js`
- **Risk**: Any throw in plan batch chain (handoff filter, recordStructuredToolEvidence, etc.) escapes the main loop, leaving session in indeterminate state
- **Root cause**: `executePlan` → `runPlanActions` → `applySettledPlanResults` → `applyAgentHandoffToSession` → `handoff-input-filter.js:112` — no try/catch on any level. Single-action path has full error boundary; plan path has none.
- **Fix**: Wrap `executePlan` call at `action-loop-session-loop.js:406` in try/catch converting throws to `plan_execution_error` observation. Also guard filter call in `handoff-input-filter.js` like `onToolResult`/`onPlannerDecision` hooks.

### C4 — Silent Compaction Failure
- **File**: `src/session/compaction.js` (catch block ~line 300)
- **Risk**: LLM compaction failure is invisible — no step, no log. Caller only sees PROMPT_BUDGET_EXCEEDED with no root cause.
- **Root cause**: Empty catch body. `action-history-compaction.js:122-134` has the correct fallback + emit pattern.
- **Fix**: Emit `session.pushStep('compaction-failed', { error: error.message })` inside catch. Mirror action-history-compaction.js fallback pattern.
- **✅ RESOLVED (AGRUN-461, 2026-06-11)**: `prepareProviderSessionContext` is session-layer and has **no `session.pushStep`** — the literal fix above does not apply there. Implemented per the existing layering (compaction.js returns structured data; handle.js translates to `onStep`): the catch now (1) **re-throws `AbortError`** so caller cancellation propagates instead of degrading into a budget error (parity with C3), and (2) captures every other error as `compactionError` on both terminal result shapes. `handle.js` emits a **`compaction-failed`** step from `prepared.compactionError`, symmetric with `compaction-completed`. Two-door parity: `approval-resume.js` forwards `compactionError` on all three compaction-running bags. Pinned by `test/unit/compaction-failure-surfaced.test.js`.

### H7 — store-memory.js Sessions Never Evicted — **✅ RESOLVED (AGRUN-488)**
- **File**: `src/session/store-memory.js`
- **Risk**: Multi-session server deployments accumulate all session data for process lifetime
- **Fix**: Add optional TTL and/or max session count with LRU eviction
- **Resolution**: `createInMemorySessionStore(options)` now accepts **opt-in** `{ maxSessions, sessionTtlMs }` (both optional; with neither set the store stays **unbounded** = current behavior, zero surprise data loss for browser / single-process hosts — chosen deliberately because evicting a session drops its history/memory, which is data loss when the in-memory store is the only source of truth). **LRU + TTL** per the chosen strategy: a per-session `{ at, seq }` access record drives both — `seq` is a strictly-monotonic counter the LRU orders by (Date.now() alone ties many same-millisecond sessions and would degenerate LRU into insertion order — a real correctness bug the test surfaced), `at` is the wall-clock the TTL idle-expiry measures. Eviction is **lazy + insert-driven, no background timer** (never holds a Node process alive): TTL expires on read (`getSession` returns null for an idle-past-TTL session) and is swept opportunistically on `createSession` (reclaims abandoned/never-re-read sessions); LRU evicts the least-recently-used on `createSession`/`saveSession` insert when the cap is exceeded; any per-session op (`appendMessage`/`writeSummary`/`appendMemory`/reads) refreshes recency. Eviction drops **all** of a session's per-session data (record + messages + summaries + memory); cross-session `globalMemory` is never evicted. Single-locus (only the in-memory store has the RAM leak; IndexedDB/FsStorage are disk-backed with their own lifecycle — no two-door). Wired through `createSessionStore(options)` (the in-memory fallback now receives `options` like the indexedDB/resilient branches already do) and exported for explicit host use; documented in `host-deployment-profiles.md` (Node server profile). Invalid `maxSessions`/`sessionTtlMs` throw a clear error. Pinned by `session-store-eviction.test.js` (5 cases: default-unbounded, LRU-not-FIFO with touch-keeps-old, eviction-drops-all-data, TTL read+sweep+active-spared, invalid-opts-throw; neuter-verified — disabling `enforceLru` makes the LRU case FAIL).

### H8 — Planner Envelope Repair N+1 + Dead MAX_RECOVERY_RETRIES — **✅ RESOLVED (AGRUN-486)**
- **File**: `src/runtime/planner.js:441`, `src/runtime/planner-recovery.js`
- **Risk**: Every invalid planner response unconditionally spawns second LLM call. MAX_RECOVERY_RETRIES is dead code (never imported/called).
- **Fix**: Check per-session repair counter before calling requestPlannerEnvelopeRepair; activate isRecoveryBudgetExhausted from planner-recovery.js
- **Resolution**: Root cause was a **vestigial parallel mechanism**. The recovery budget (`recoveryState.retries` / `MAX_RECOVERY_RETRIES` / `noteRecoveryAttempt` / `isRecoveryBudgetExhausted`) was built (ADR-0014) to bound exactly this N+1 repair call, but went dead two ways: (1) `action-loop-session-cycle.js` reset `retries=0` at the **start of every cycle** so it was always 0 at planner time → `isRecoveryBudgetExhausted` could never fire; (2) nothing ever incremented it. Meanwhile the live `plannerInvalidCount` counter did the invalid-cycle bookkeeping. The fix revives the budget as a true **repair-spend gate** rather than deleting the public `result.recoveryState` telemetry (which was always-zero and now reports a meaningful streak): `planner.js` now gates `requestPlannerEnvelopeRepair` behind `isRecoveryBudgetExhausted(runState)` (after `MAX_RECOVERY_RETRIES`=2 consecutive attempted-and-failed repairs, it emits `planner-repair-budget-exhausted` and surfaces the null decision so the planner re-plans and terminal-repair/invalid-action-convergence own recovery); `action-loop-failure.js` ticks the budget via `noteRecoveryAttempt` **only when a repair was actually spent** (`plannerResult.repairAttempted === true`); `action-loop-planner.js` clears it via the new SSOT `resetRecoveryBudget(runState)` the moment a valid decision is produced (both valid-decision sites), which preserves ADR-0014's "don't starve a long session after one bad turn" intent without the dead per-cycle reset. The genuinely-dead `buildRecoverySignal` + `buildStrictRetryPrompt` (+ their private-only helpers) were deleted. **Dispatch-path note**: the N+1 repair LLM call exists ONLY on the envelope planner path (native-tools mode makes no runtime repair call — it returns `repairAttempted:false`), so the gate is wired at its single locus; no plan/single two-door split applies (the planner runs once per cycle above both action dispatch paths). Pinned by `planner-repair-budget-gate.test.js` (real-probe via an injected transport that counts every provider completion: exhausted budget → 1 call + budget-exhausted step; fresh budget → 2 calls; increment ticks only on attempted-and-failed repair — verified to FAIL, 2 calls instead of 1, when the gate is neutered) + `recovery-budget.test.js` (resetRecoveryBudget cases).
