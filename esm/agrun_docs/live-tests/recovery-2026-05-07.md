## Goal

Verify that ADR-0014's three-PR refactor (delete English-only regex →
add bounded budget + diagnostics → ship recovery_signal hint in
planner prompt) actually closes the Mandarin recovery hole identified
in the AGRUN-220 4-cell matrix (1/4 usable cells, Mandarin × any model
silently failed at recovery).

Success target: ≥3/4 cells recover from a single induced
invalid-envelope without the run terminating in error, and
`diagnostics.recovery.retries ≤ 2` in every run.

## Verification procedure

1. Build dist + start browser example dev server (`npm run example:dev`).
2. Open Chrome via Chrome DevTools MCP at `http://localhost:5173`.
3. For each cell:
   1. Clear IndexedDB stores `agrun-browser-chat-store` and
      `agrun-browser-runtime-session-store` between cells (otherwise
      qa session config persists).
   2. Configure provider+model+language via `?qa=…` URL params.
   3. Paste prompt; let agent run.
   4. Capture: cycle count, last `diagnostics.recovery` snapshot,
      planner step trace, final answer (or termination reason).
4. Aggregate into the table below and commit.

## Failure-mode plan

If C2 / C4 still fail after PR 3:
1. Check `result.diagnostics.recovery.lastSignal` — is the runtime
   even classifying the failure as a recovery event?
2. Check planner-prompt rendered output — is the new
   `loopState.recoveryContext` line surfacing on the retry turn?
3. If runtime classifies but model ignores: not an ADR-0014 bug; SKILL.md
   needs hardening.
4. If runtime never classifies: bug in PR 1's
   `shouldFallbackToWebSearch` simplification — restore the action +
   pending-clarification gate logic and reopen the ticket.
