# Planner Architecture

> **Last reviewed:** 2026-05-19 against ADR-0023..0033.
>
> Sections marked **HISTORICAL** describe push-mode behavior deleted by [ADR-0023](./adr/0023-harness-as-tool-provider-only.md) (5/8) and [ADR-0026](./adr/0026-zero-residual-push-mode.md) (5/9). The runtime no longer:
> - runs a strict-retry 3rd API call (Stage 3 below)
> - applies `applyPlannerGuardrails` / `finalize_to_clarify` / `clarify_to_finalize` rewrites (Planner Guardrails section below)
> - vetoes AI's finalize via `final-response-quality.js` or `research-coverage-guard.js`
>
> When AI's output is bad, AI's output is bad. The harness exposes signals (`runState.actionFailureSignal`, `runState.finalResponseQuality.lastIssues`, `runState.sessionBudget`); AI decides next move. See [ADR-0023](./adr/0023-harness-as-tool-provider-only.md) for full list.

## Purpose

This document describes the planner system that drives action selection in the agrun.js action loop. The planner is the LLM-backed decision engine that chooses what action to take next during multi-cycle execution.

For the action contract and registry, see `agrun_docs/action-contract.md`.
For the full execution flow, see `agrun_docs/agentic-execution-flow.md`.

## Overview

The planner sits at the DECIDE phase of the OODAE cycle. It receives the current execution context and returns a structured decision envelope telling the runtime what to do next.

```text
Context + Available Actions
        ↓
   Planner Prompt Construction
        ↓
   Provider Completion Request
        ↓
   Envelope Parsing
        ↓
   Repair Cascade (if invalid)
        ↓
   Decision Envelope
```

## OODAE Phase Mapping in Envelope Mode

Each envelope call completes a full OODAE cycle in a **single LLM emission**. Runtime does not split cognition across multiple API calls by phase. See [ADR-0033](./adr/0033-oodae-as-single-call-structured-emission.md) for the full rationale.

| OODAE Phase | How it appears in envelope mode | Owner |
|---|---|---|
| **Observe** | Runtime projects `toolContext.lastResult`, `readSources`, `virtualWorkspace`, `observation`, `loopState.*` into planner prompt | Runtime (mechanism) |
| **Orient** | `reasoning` field — AI writes its rationale before choosing action | AI (policy) |
| **Decide** | `type` + `name` + `args` — the chosen next step | AI (policy) |
| **Act** | Runtime executes chosen tool or terminal handler; no LLM call | Runtime (mechanism) |
| **Evaluate** | `finalReadiness.requirementsAssessment` on `finalize`; implicit in `reasoning` on non-terminal actions | AI (policy) |

This is structurally equivalent to ReAct (reasoning + acting + observation interleaved in one call). Envelope mode is richer than native_tools here — `reasoning` and `requirementsAssessment` are enforced by schema, while native tool_use has no equivalent fields.

**The OODAE loop is NOT 5 LLM calls per cycle.** Splitting by phase would require runtime to slice AI's cognitive context, which is push-mode and multiplies cost/latency without benefit. See ADR-0033 for the rejected alternative.

## Decision Envelope Contract

The planner returns one of three decision types:

### Action Decision

```json
{
  "type": "action",
  "name": "web_search",
  "args": { "query": "..." },
  "reasoning": "..."
}
```

Tells the runtime to execute the named action with the given arguments.

### Clarify Decision

```json
{
  "type": "clarify",
  "question": "...",
  "reasoning": "..."
}
```

Tells the runtime to ask the user for clarification before proceeding.

### Final Decision

```json
{
  "type": "final",
  "answer": "...",
  "reasoning": "..."
}
```

The planner provides a direct answer without executing any further actions.

### Finalize Decision

```json
{
  "type": "finalize",
  "instruction": "...",
  "reasoning": "..."
}
```

Tells the runtime to synthesize a final answer from gathered evidence via a dedicated provider call. It can be selected by the planner when evidence is sufficient, or generated by guardrails / continuity resolution.

## Prompt Construction

`buildPlannerPrompt()` assembles the planner prompt from:

- **User prompt**: The original user request
- **Available actions**: List of actions the planner can choose from, with descriptions and argument schemas
- **Action history**: What actions were taken in prior cycles and their outcomes
- **Planner state**: Current goal, topic, active query, evidence state, clarification status
- **Research context**: Search results, read sources, verification state
- **Agent skill context**: Active skill and tool execution history
- **Skill catalog with arg hints**: Compact parameter schema summaries for each bundled tool

### Skill Catalog Arg Hints

`toSkillCatalogSummary()` produces a compact catalog entry for each bundled agent skill. Each tool in the catalog includes an `args` hint derived from the tool's parameter schema:

```json
{
  "name": "list_purchase_orders",
  "description": "List purchase orders with optional filters",
  "args": { "filters": "object{party_desc:string,party_code:string,amount_sign:string|positive|negative}", "sortDir": "string|ASC|DESC", "pageSize": "number" },
  "required": ["filters"]
}
```

The `args` field maps parameter names to compact type hints. Enum-constrained fields append allowed values as `type|value1|value2`, and object-type parameters with nested properties use `object{key:type,...}` notation. This lets the planner see both valid sub-keys and valid enum values, reducing guesses such as `limit` vs `pageSize`, flat `party_name` vs `filters.party_desc`, or invalid grouping values.

When no skill is active yet, the catalog remains compact, but tools with enum-constrained parameters include a lightweight `toolArgHints` entry so the planner can still see allowed values without receiving full tool descriptions.

`buildPlannerSystemPrompt()` provides the system-level instructions:

- Role definition (decision-making agent)
- Per-run dynamic system prompt from `request.systemPrompt` (e.g. language instructions, persona overrides)
- Envelope format requirements (JSON)
- Available action descriptions with argument schemas
- Constraints on when to clarify vs finalize

### Terminal Envelope Policy

`buildEnvelopeLines()` is the single source of truth for terminal envelope examples shown to the planner and repair request. (The strict-retry path was deleted in ADR-0023.)

By default, agrun keeps both terminal shapes available:

- `type:"finalize"` — runtime performs a dedicated final synthesis call from gathered evidence.
- `type:"final"` — planner directly returns the user-facing answer.

One narrow high-risk context is filtered: when the effective planner mode is `envelope`, the request is a Gemini lite model, and TodoState still has active or pending items, `type:"finalize"` is omitted from the valid-envelope examples and repair guidance says to use `final`. If the planner still returns `finalize` in that context, envelope validation rejects it before execution so the host invalid-output hook or repair cascade can recover. This prevents lightweight models from mirroring repeated `finalize` envelopes during unfinished TodoState loops, while preserving normal runtime-finalize evidence synthesis everywhere else.

### Dynamic System Prompt Injection

`request.systemPrompt` is injected into both planner paths (envelope mode and native tools mode). The injection sits after the role block but before the planner's core rules:

```text
systemInstruction = [
  roleBlock (ROLE.md)              ← static, from createRuntime({ role })
  request.systemPrompt             ← dynamic, from runInput.systemPrompt
  BASE_SYSTEM_LINES                ← planner JSON/envelope rules
  envelope/action definitions      ← action schemas
]
```

This ensures that per-run instructions (such as response language, tone, or persona overrides) take effect on all response paths — including `type: "final"` direct answers that bypass the finalizer.

When `systemPrompt` is empty or not provided, the planner system prompt is identical to the pre-injection behavior.

### Caller-Supplied `plannerDirectives`

`runtimeConfig.plannerDirectives: string[]` is appended **after** all built-in lines, so the effective prompt is:

```text
systemInstruction = [
  roleBlock
  request.systemPrompt
  BASE_SYSTEM_LINES
  envelope examples + action descriptions + per-action guidance
  built-in rules (list_agent_skills, execute_skill_tool, use_agent_skill, "prefer finalize on lastResult", finalize/final)
  plannerDirectives                ← caller-supplied, override-by-recency
]
```

Because LLMs weight later instructions more heavily, directives appended here effectively override earlier built-in rules without editing runtime source.

### Toggling the "prefer finalize on lastResult" Rule

The built-in line "If the needed skill tool result is already present in `toolContext.lastResult`, prefer finalize instead of repeating the same `execute_skill_tool` call." conflicts with legitimate multi-step / multi-doctype / multi-entity plans. Two controls:

- `runtimeConfig.preferFinalizeOnLastResult: false` — drop the line entirely.
- Return `resultKind: "resolution"` from tools that perform lookup/identifier resolution. When present, the built-in line carries an exception sentence telling the planner to continue with the next evidence-gathering call. The runtime also skips the single-tool fast path on resolution results.

## Planner State

`plannerState` is derived from the run state and context snapshot before each planner call:

```text
PlannerState
 ├ currentGoal        — extracted goal from user prompt
 ├ currentTopic       — extracted topic
 ├ activeQuery        — current search/research query
 ├ evidenceState      — none | partial | sufficient
 ├ ambiguityState     — clear | moderate | high
 ├ goalQuality        — stable | unstable
 ├ pendingClarification — structured clarification object or null
 ├ openAmbiguity      — text describing what's still unclear
 ├ promptSignal       — low | medium | high (prompt complexity)
 └ directToolCandidate — candidate for direct tool bypass
```

## Repair Cascade

When the planner returns an invalid or unparseable response, the runtime attempts a three-stage repair:

### Stage 1: Soft Repair (`repairPlannerEnvelope`)

- Attempts to parse the response as JSON via [`parseLooseJsonValue()`](https://github.com/yapweijun1996/agrun/blob/main/0_development/src/runtime/semantic-json.js). Extraction uses a depth-aware, string-escape-aware balanced-bracket scanner (`extractBalancedJson()`). Unclosed or partially-written JSON returns `null` rather than being salvaged into a malformed payload, and a response containing two distinct objects yields the first balanced match only — `{"a":1} and then {"b":2}` resolves to `{"a":1}` instead of swallowing the interstitial text.
- Applies structural corrections (missing fields, type coercion)
- Handles common format deviations (markdown wrappers, trailing text)
- **Plan action tolerance:** `normalizePlanActions()` skips individual actions that fail shape or name validation (emitting an `onInvalidAction(reason, record)` event) instead of discarding the whole plan. A five-step plan with one misnamed action still executes the four valid steps rather than forcing a full re-plan round-trip.
- **Required skill/tool fields guard:** If the planner requests `execute_skill_tool` without `skillName` AND without `toolName`, and no active skill exists, the envelope is rejected (returns null). This triggers the repair cascade so the model retries with the correct fields. Without this guard, inference fallback would attempt arg-key matching across all bundled tools, which cannot reliably distinguish tools with identical parameter schemas (e.g. 51 tools sharing `filters/sortField/sortDir/pageSize`)
- **Reusable tool result guard:** If the planner requests `execute_skill_tool` and `toolContext.lastResult` already exists, forces `finalize` to prevent infinite loops. Triggers both when `toolName` matches the last result and when `toolName` is omitted (the action would resolve to the same tool via fallback inference). **Exception:** when the last result is an error (`result.ok === false` or `result.error` present), the guard is bypassed to allow self-correction retries with corrected arguments

### Stage 2: Envelope Repair Request (`requestPlannerEnvelopeRepair`)

- Sends the invalid response back to the provider with explicit format instructions
- Preserves the original planner system instruction as the repair call's system-level context, then appends the repair-specific instruction block
- Uses the same terminal envelope policy as the main planner prompt, so high-risk contexts that hide `finalize` do not reintroduce it during repair
- Asks the provider to reformat its response into a valid envelope
- Parses the repair response

### Host Invalid-Output Hook (`onInvalidPlannerOutput`)

Before Stage 2, hosts may install `onInvalidPlannerOutput(rawText, parseError, runState)`. If the hook returns a valid planner envelope, the runtime uses it and skips the repair LLM call. If it returns `null` / `undefined`, throws, or returns an invalid envelope, the normal repair cascade continues.

### ~~Stage 3: Strict Retry~~ (HISTORICAL — deleted in ADR-0023)

The strict-retry 3rd API call (`buildStrictRetryPrompt`) was deleted in [ADR-0023](./adr/0023-harness-as-tool-provider-only.md) (5/8). It was push-mode: runtime authored a hardcoded "stricter" prompt asking the LLM to retry with extra constraints. When envelope repair (Stage 2) returns null, the null decision now surfaces — AI re-plans next cycle on its own.

### Stage 3: Fallback to Web Search (bounded, non-research only)

If repair fails and conditions are met:

- `shouldFallbackToWebSearch()` checks active goal/query, `web_search` availability, no pending clarification.
- `isResearchQualityGateRequired` and `isRecoveryBudgetExhausted` block the fallback when long-research is active or recovery budget (2 retries) is spent.
- Creates a `web_search` action decision with an inferred query.

Note: this remains the last bounded push-mode site (V1 in the non-ai-first audit). It only fires for non-research simple traffic and is bounded by a 2-retry budget. Long-term goal is to delete it entirely.

```text
Planner Response
    ↓
Parse Envelope → valid? → use decision
    ↓ (invalid)
Soft Repair → valid? → use decision
    ↓ (invalid)
onInvalidPlannerOutput? → valid? → use decision
    ↓ (invalid)
Envelope Repair Request → valid? → use decision
    ↓ (invalid)
Fallback to web_search (non-research, within budget)? → yes → use fallback
    ↓ (no)
Return null → AI re-plans next cycle
```

## ~~Planner Guardrails~~ (HISTORICAL — deleted 2026-05-09)

`applyPlannerGuardrails` and the `finalize_to_clarify` / `clarify_to_finalize` decision rewriters were deleted on 2026-05-09 (see in-source tombstone at [action-loop-session-loop.js:200](https://github.com/yapweijun1996/agrun/blob/main/0_development/src/runtime/action-loop-session-loop.js)). If AI emits unsafe finalize or unnecessary clarify, AI sees the consequences via observation/turn outcomes and re-plans.

## Planner Modes

agrun.js supports two planner decision encodings. The default host-facing value is `plannerMode: "auto"`, where agrun resolves the effective mode from provider/model/tool-surface capabilities. Explicit `"native_tools"` and `"envelope"` remain advanced/debug overrides.

### Auto Mode (Default)

```javascript
createRuntime({ plannerMode: "auto", ... })
```

Auto mode uses `src/runtime/provider-capabilities.js` as the SSOT for planner-mode resolution:

```text
configuredMode → resolvePlannerMode() → effectivePlannerMode
```

Since ADR-0058 (2026-07-02, supersedes ADR-0031), the resolver still inspects nothing but configuredMode. `plannerMode: "auto"` (and any omitted / unknown value) resolves to `effectiveMode: "native_tools"` with `reason: "default_native_tools"`. Explicit `plannerMode: "native_tools"` and `plannerMode: "envelope"` are honored with `reason: "explicit"`. Envelope remains fully supported as the explicit opt-out (weak models, ADR-0035 envelope prompt-override users) and as the automatic per-call fallback via `nativeToolsFailurePolicy: "fallback_to_envelope"` (default).

### Envelope Mode

```javascript
createRuntime({ plannerMode: "envelope", ... })
```

The LLM returns a JSON text envelope that the runtime parses:

```text
LLM → JSON text → parsePlannerEnvelope() → repair cascade → decision
```

Advantages:
- Works with any text-capable LLM
- Full repair cascade for malformed responses
- Deterministic guards and envelope normalization

### Native Tools Mode

```javascript
createRuntime({ plannerMode: "native_tools", ... })
```

The LLM uses the provider's native tool/function calling API to select actions:

```text
Actions → buildOpenAITools() / buildGeminiTools() → provider API
LLM → tool_call response → parseToolCallDecision() → decision
```

Advantages:
- Provider validates tool calls natively (type checking, required args)
- Better semantic understanding by the LLM (sees structured schema)
- Reduced prompt overhead (no envelope examples needed)
- No loose JSON repair needed
- Alignment with standard LLM patterns (OpenAI function calling, Gemini function declarations)

### Native Tools Auto Readiness

ADR-0031 (2026-05-16) supersedes AGRUN-213q. `plannerMode: "auto"` now resolves to `envelope` for every provider/model/action surface; native tools is an explicit opt-in advanced/debug mode only:

- envelope is now the canonical PASS path; live-verified twice in a row on OpenAI `gpt-5-mini` for the canonical 3000-word long-research scenario (see `agrun_docs/live-tests/node-agrun-3000-double-baseline-2026-05-16.md`)
- native tools still support single action, `ask_clarification`, `finalize`, `final_answer`, and `plan`, and native `plan` still parses into the same `type:"plan"` decision shape that reuses `validatePlan()` / `executePlan()`
- broad native-readiness live coverage previously passed OpenAI/Gemini native action, clarify, finalize, approval, search, and TodoState, but Gemini native preview endpoints were observed to be unstable under sustained long-running native requests on 2026-05-16, which is the primary driver for the envelope default
- native planner failures still default to `nativeToolsFailurePolicy: "fallback_to_envelope"` when a host explicitly opts into native; `"hard_fail"` is available when hosts want native-mode failure to stop immediately
- debug surfaces still expose configured mode (`auto`/`envelope`/`native_tools`), effective mode, and resolver reason (`default_native_tools` / `explicit`) so hosts can observe agrun's choice

The readiness matrix and blocker list live in [native-tools-readiness.md](./native-tools-readiness.md). The default-mode decision is recorded in [ADR-0003](./adr/0003-native-tools-default-readiness.md).

### Native Tools Failure Policy

`native_tools` mode is not infallible — providers can return schemaless errors, tool-call payloads that don't match any declared action, or generic 5xx responses. The default policy is compatibility-first:

```javascript
createRuntime({
  plannerMode: "auto",
  nativeToolsFailurePolicy: "fallback_to_envelope"
});
```

When the policy is `"fallback_to_envelope"`, `planNextAction()` falls back to envelope mode rather than aborting:

```text
effectivePlannerMode === "native_tools"
    ↓
try planNextActionWithNativeTools()
    ├─ success → return decision
    ├─ CIRCUIT_OPEN → surface circuit-open result (no fallback)
    └─ other error
        ↓
        emit step: "planner-native-tools-fallback" (error message normalized)
        ↓
        try planNextActionWithEnvelope()
            ├─ success → return decision
            ├─ CIRCUIT_OPEN → surface circuit-open result
            └─ error
                ↓
                envelopeError.nativeToolsCause = <original native_tools error>
                throw envelopeError
```

Key points:

- **Both root causes are preserved.** When both modes fail, the re-thrown error carries the envelope failure as its own `.message` and the original native_tools failure on `.nativeToolsCause`. Downstream handlers can inspect both:
  ```js
  try { await runtime.run(input); }
  catch (err) {
    console.log(err.message);            // envelope failure
    console.log(err.nativeToolsCause);   // original native_tools failure
  }
  ```
- **Step trace captures the fallback transition.** The `planner-native-tools-fallback` step is emitted with the normalized native_tools error message so live observers know fallback is happening even before the envelope attempt finishes.
- **No fallback on `CIRCUIT_OPEN`.** The circuit breaker already signals "provider is unhealthy, don't retry" — falling back to envelope against the same dead provider would just produce a second failure.
- **Hard-fail is explicit.** With `nativeToolsFailurePolicy: "hard_fail"`, the runtime emits `planner-native-tools-failed` and returns the normalized planner/provider error without trying envelope mode. This is useful for provider-mode debugging and host surfaces that prefer strict native behavior.

### Tool Definition Conversion

`planner-tools.js` converts action registry definitions to provider-native tool formats:

| agrun Action | OpenAI Tool | Gemini Tool |
|-------------|-------------|-------------|
| `web_search` | `{ type: "function", name: "web_search", parameters: {...} }` | `functionDeclarations: [{ name: "web_search", parameters: {...} }]` |
| `ask_clarification` (clarify) | `{ type: "function", name: "ask_clarification", ... }` | Same, converted to Gemini schema |
| (finalize) | `{ type: "function", name: "finalize", ... }` | Same |
| (final) | `{ type: "function", name: "final_answer", ... }` | Same |

Parameter schemas prefer the action registry's planner `argsSchema`; `argsExample` is only a fallback for legacy action definitions. Gemini tool schemas add recursive `propertyOrdering` so required/control keys are emitted before nested payload fields. The native `plan` tool uses the same plan validation/execution harness as envelope `type:"plan"` decisions. For `execute_skill_tool` inside native `plan`, providers may use flatter compatibility shapes: `skillName` / `toolName` / `toolArgs`, `toolArgsJson`, `arg_*`, or direct flat tool-specific fields. The parser normalizes each shape back to canonical `args.skillName` / `args.toolName` / `args.args` before validation. For Gemini live readiness, `toolArgsJson` is the required compatibility shape today; nested `toolArgs` and direct flat fields remain provider-risk paths. Provider-specific native-plan guidance comes from `provider-capabilities.js`, keeping prompt text and schema descriptions aligned.

When runtime debug logging is enabled, native planner calls log only a scrubbed raw argument shape: tool name, keys, primitive types, string lengths, array lengths, and sensitive-key counts. Raw argument values and sensitive fields are not logged.

## Action Surface Filtering

Before the planner prompt is built, `selectPlannerActions()` (`src/runtime/planner-action-surface.js`) filters the list of available actions shown to the LLM. The filtering has a strict priority order:

```text
All registered actions
    ↓
1. hard_veto convergence filters (always applied first — override everything)
    ├─ readOnlyPlanningState hard_veto
    └─ workspaceMutationGrowthConvergence hard_veto
    ↓
2. terminalRepairState.allowedActions (if terminal repair is active)
    → show ONLY the allowedActions; all others hidden
    ↓
3. exactAction / structureRepair convergence filters (advisory-equivalent surface filters)
    ↓
Available action surface shown to LLM
```

### Priority Rule: hard_veto > terminalRepairState.allowedActions

`terminalRepairState.allowedActions` narrows the action surface to a recovery-focused subset (e.g. `["workspace_write", "workspace_replace", "workspace_publish_candidate"]`). However, **hard_veto convergence signals override this allowlist**. When a convergence detector (e.g. `readOnlyPlanningState`, `workspaceMutationGrowthConvergence`) reaches `escalation: "hard_veto"`, its `forbiddenActions` are removed from the surface **before** the allowlist early-return is applied.

Advisory-escalation convergence signals (`escalation: "advisory"`) do NOT override the allowlist — they communicate pressure via the planner prompt context only.

This is enforced in two places:

| Location | Role |
|----------|------|
| `selectPlannerActions` (planner-action-surface.js) | Hard_veto forbidden actions filtered from LLM surface first |
| `maybeBlockReadOnlyPlanningLoop` (action-loop-action.js) | Hard_veto preflight: even if action is in repairAllowedActions, still blocked if in hard_veto forbiddenActions |

### Convergence Surface Filters

| Convergence State | Activation | Surface Effect |
|-------------------|------------|----------------|
| `exactAction` convergence | `forbiddenMove: "repeat_same_action_args"` for a specific `actionName` | Hides that specific action |
| `readOnlyPlanningState` advisory | `ignoredCount ≥ 1`, `escalation: "advisory"` | No surface change (prompt-level pressure only) |
| `readOnlyPlanningState` hard_veto | `ignoredCount ≥ 3`, `escalation: "hard_veto"` | Hides all `forbiddenActions` — overrides terminalRepair allowlist |
| `structureRepairConvergence` | `active: true` | Hides `forbiddenActions` (read/list actions during structure repair) |
| `workspaceMutationGrowthConvergence` advisory | `stallCount ≥ 2`, `escalation: "advisory"` | No surface change |
| `workspaceMutationGrowthConvergence` hard_veto | `stallCount ≥ 3`, `escalation: "hard_veto"` | Hides all `forbiddenActions` — overrides terminalRepair allowlist |

### workspaceMutationGrowthConvergence: write-only tracking

`updateWorkspaceMutationGrowthConvergence` tracks workspace mutation stalls to catch write→overwrite oscillation. The stall counter only advances on `workspace_write` actions; other mutations (`workspace_append`, `workspace_replace`, `workspace_insert`) pass through without resetting the counter. A destructive overwrite (`deltaWords < 0`) activates advisory immediately without waiting for the 2-stall threshold.

**Why write-only**: if append/replace positive-delta resets the stall counter, a write→append→write oscillation is masked indefinitely. Tracking only write stalls keeps the signal clean.

## Session Context Projection

Before calling the planner, the runtime projects session context from the context snapshot:

```text
contextSnapshot
    ↓
projectSessionContextFromSnapshot()
    ↓
projectSessionContextFromPlannerState()
    ↓
sessionContext (passed to provider)
```

This provides the planner with:

- Current goal and topic
- Clarification status and history
- Open ambiguity
- Last resolution details

The session context is passed alongside the planner prompt to give the provider conversational continuity.

## Why agrun.js Does Not Use a Mode Router

A common pattern in the agent literature is the **Mode Router**: an upfront classifier that picks one of {ReAct, Plan-and-Solve, ReWOO, LLMCompiler, LATS, Self-Discover} based on task type, then runs that mode end-to-end.

agrun.js intentionally does NOT use a Mode Router. The reasons:

1. **Modes are not mutually exclusive — they are loop phases.** A real run reasons (ReAct), plans (Plan-and-Solve via TodoState), gathers evidence (ReWOO-style with the evidence graph), and reflects (`final-response-quality.js`) — all in the same OODAE loop. A static router would force one behavior and disable the others.

2. **Static dispatch is hardcode thinking.** A router is an `if/else` over task type. agrun.js prefers letting the planner adaptively choose actions inside a unified loop; this is the harness-engineering principle (let the system shape behavior, do not pre-declare it).

3. **Browser-side cost.** Each routing decision adds an LLM call before any work begins, doubling first-token latency for no observable benefit. Task classification is already implicit in the planner's first decision.

4. **LLMCompiler / LATS do not pay off in this environment.** DAG parallelism is bottlenecked by provider rate limits, not by local CPU. Tree search multiplies token cost N-fold against a chat UI where users will not wait. Both are excluded by design, not by oversight.

5. **Self-Discover is implicit.** The planner system prompt already steers behavior by task shape (research vs coding vs chat). Adding an explicit Selector + Adaptor + Implementor layer would be a three-call detour for an outcome the planner already achieves in one call.

What agrun.js DOES use:

- **OODAE loop** = ReAct (reasoning + acting + observing interleaved)
- **TodoState + autopilot** = Plan-and-Solve (structured decomposition before tool use)
- **Evidence graph + evidence pack** = ReWOO (decoupled evidence gathering)
- **`final-response-quality.js` + `research-coverage-guard.js`** = Read-only signal reflection (post-ADR-0023: results surface to AI via `runState.finalResponseQuality.lastIssues`; runtime no longer vetoes)
- **Session memory + tier policy** = bounded Reflexion (lessons learned without unbounded memory growth)

These are **fused into one loop**, not selected by a router. New patterns from the literature should be evaluated against this loop first: if the capability is already implicit, do not add a new module just to name it.
